ARCHITECT · v1architecture.md

Approach

Expose readiness via a dedicated HealthController at /v1/health rather than overloading Spring Actuator's /actuator/health, so the product owns the contract and response shape. Liveness stays on Actuator; readiness aggregates downstream dependency probes.

Components

  • HealthController — REST entrypoint, extends BaseController, returns Optional<HealthStatus>
  • HealthService — aggregates probes (Mongo, Vault, Arbiter gRPC), extends BaseService
  • HealthStatus — immutable record: status, checks[], timestamp

Contract

# GET /v1/health → 200 (ready) | 503 (degraded) { "status": "UP", "checks": [ { "name": "mongo", "status": "UP" }, { "name": "vault", "status": "UP" }, { "name": "arbiter", "status": "UP" } ] }

Constraints

  • No new module — lands in service-health under existing layered rules
  • Probe timeout 2s per dependency; degrade open (report DOWN, never block)
  • Open question routed to SECURITY: should /v1/health require @RequireScope?
